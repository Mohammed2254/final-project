# Stage 2: Project Charter

**Project:** Farah — Wedding Planning Platform
**School:** Holberton School — Foundations Track, Semester 3
**Date:** June 13, 2026


## 1. Project Objectives

### Purpose

Farah is a digital platform that connects engaged couples with wedding service providers in one place. It enables individual or collaborative planning between partners, saving time and effort in organizing wedding preparations.

### Objectives

**Objective 1:** 
- Build a platform that allows couples to browse and book at least two service categories — wedding halls and photographers, within four weeks of the development stage.

**Objective 2:** 
- Enable service providers to register their offerings, set pricing, and manage incoming requests through a dedicated vendor dashboard by the end of Stage 4.

**Objective 3:** 
- Deliver a shared couple workspace that allows both partners to make booking decisions together in the same session by the end of the second month of development.

## 3. Scope

### In Scope

- User registration and authentication for couples and service providers
- Browsing and viewing service listings
- Booking services individually or as a complete package
- Shared couple workspace for joint planning and decision-making
- Saving favorite services for later review
- Vendor dashboard for managing listings and incoming bookings

### Out of Scope

- Mobile application
- Real online payment processing
- Folk music bands, jewelry, catering hospitality, makeup artists, and gift delivery
- Services outside Saudi Arabia


## 4. Risks

**Risk 1 — Time pressure with a part-time team**
- Mitigation: Design the MVP so that it can be completed by two people in the worst case, ensuring the project is never blocked by a single member's availability.

**Risk 2 — Scope creep**
- Mitigation: Freeze the scope after Stage 2. Any new feature requests are logged and deferred to a future version.

**Risk 3 — Shared workspace complexity**
- Mitigation: Implement a simple polling mechanism first to sync couple sessions, or other preferable tools. Upgrade to real-time sync only if time and resources allow in Stage 4.

**Risk 4 — Team not meeting timeline commitments**
- Mitigation: Weekly standups with explicit deadlines assigned per task and progress tracked on GitHub.

**Risk 5 — Technical disagreements on stack or approach**
- Mitigation: Final technical decisions rest with the Team Lead. Discussions happen before implementation, not during it.
